## Q1. 개인정보(전화번호, 사업자번호)를 AES-256으로 암호화하는 이유는?

**A:** ISMS(정보보호 관리체계) 인증 기준과 개인정보보호법에서 개인정보의 암호화 저장을 요구합니다.

DB가 유출될 경우:
- **평문 저장:** 전화번호, 사업자번호가 그대로 노출 → 법적 제재 + 신뢰도 하락
- **AES-256 암호화:** 암호문만 노출 → 키가 없으면 복원 불가

BCrypt가 아닌 AES-256을 쓰는 이유:
- 전화번호/사업자번호는 **조회 시 원문이 필요**합니다 (화면에 표시해야 하니까).
- BCrypt는 단방향이라 복호화 불가 → AES-256 양방향 암호화가 적합합니다.

```java
// 저장 시: 평문 → 암호문
customer.encryptFields(AES256Util.encrypt(phone), AES256Util.encrypt(businessNumber));

// 조회 시: 암호문 → 평문
dto.setPhone(AES256Util.decrypt(dto.getPhone()));
```

## Q2. 사업자번호 유효성 검증 알고리즘을 설명해주세요.

**A:** 한국 사업자등록번호는 10자리 숫자이며, 마지막 1자리가 **체크 디짓**입니다.

```
사업자번호: 1 2 3 - 4 5 - 6 7 8 9 0
가중치:     1 3 7   1 3   7 1 3 5
```

검증 방법:
1. 각 자리 숫자와 가중치를 곱합니다.
2. 9번째 자리는 `(숫자 × 5) / 10`의 몫도 추가로 더합니다.
3. 모든 합계를 10으로 나눈 나머지와 마지막 자리를 더해 10의 배수가 되면 유효합니다.

이 검증을 **서비스가 아닌 별도 유틸 클래스**(`BusinessNumberValidator`)에 넣은 이유:
- 사업자번호 검증은 순수한 계산 로직이므로 DB나 스프링에 의존할 필요가 없습니다.
- `static` 메서드로 어디서든 호출 가능합니다.
- 단위 테스트가 쉽습니다 (스프링 컨텍스트 없이 테스트 가능).

## Q3. 암호화된 필드로 검색(LIKE)이 안 되는 문제는 어떻게 해결하나요?

**A:** AES-256 암호화된 값은 LIKE 검색이 불가능합니다.
같은 "010-1234-5678"이라도 암호화하면 매번 다른(또는 같은) 긴 문자열이 되어 패턴 매칭이 안 됩니다.

해결 방법:
1. **검색 가능 필드와 암호화 필드를 분리:** `shop_name`, `owner_name`은 평문으로 검색 가능하게 유지하고, 개인정보(`phone`, `business_number`)만 암호화
2. **정확히 일치하는 검색만 지원:** 전화번호 검색이 필요하면 입력값을 먼저 암호화한 후 `WHERE phone = #{encryptedPhone}`으로 비교
3. **해시 인덱스 컬럼 추가:** `phone_hash` 컬럼을 별도로 두고 SHA-256 해시를 저장, 검색에 활용

이 프로젝트에서는 방법 1을 채택했습니다. 고객 목록 검색은 미용실명/대표자명으로만 가능하고, 전화번호/사업자번호는 상세 조회에서만 복호화하여 표시합니다.

## Q4. Customer 수정 시 사업자번호 중복 체크를 왜 "자기 자신을 제외"하고 하나요?

**A:** 수정 시 사업자번호를 변경하지 않았는데도 중복으로 판정되는 문제를 방지하기 위해서입니다.

```java
// 잘못된 방식: 자기 자신도 중복으로 잡힘
boolean exists = repository.existsByBusinessNumber(encryptedBizNum);  // true (자기 자신!)

// 올바른 방식: 자기 자신을 제외하고 검사
boolean exists = repository.existsByBusinessNumberAndIdNot(encryptedBizNum, customerId);
```

예시: ID=1인 고객의 사업자번호가 "123-45-67890"이고, 다른 필드만 수정하는 경우
→ 자기 자신을 제외하지 않으면 "이미 등록된 사업자번호"라고 에러가 발생합니다.

## Q5. MyBatis에서 조회한 후 Service에서 복호화하는 구조의 장단점은?

**A:**

**현재 구조:**
```
MyBatis(암호문 조회) → QueryService(복호화) → Controller(평문 응답)
```

**장점:**
- MyBatis는 순수 SQL만 담당 → SQL이 단순해짐
- 복호화 로직이 Service 한 곳에 집중 → 관리 포인트가 명확

**단점:**
- 목록 조회 시 N건의 데이터를 루프 돌면서 복호화 → 성능 부담
- Service에서 복호화를 빠뜨리면 암호문이 그대로 응답될 위험

**대안:**
- MyBatis TypeHandler: `@MappedTypes`로 자동 암호화/복호화 (SQL에서 투명하게 처리)
- JPA AttributeConverter: `@Convert` 어노테이션으로 Entity 레벨에서 자동 처리

실무에서는 TypeHandler/AttributeConverter를 사용하는 것이 더 안전합니다. 이 프로젝트에서는 학습 목적으로 명시적 복호화를 사용했습니다.
